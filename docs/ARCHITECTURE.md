# 架构决策记录

> 本文档记录关键架构决策（ADR: Architecture Decision Records），
> 每一条决策都说明背景、选择、理由与替代方案。

## ADR-001: 前后端分离架构

**背景**：项目最终需对接已有决策平台，技术栈与鉴权方案均未知。

**决策**：采用前后端完全分离架构，后端只暴露 RESTful API，
前端为独立的"参考实现"，可被未来平台替换。

**理由**：
- 平台对接时只需替换前端或加 adapter 层，后端零改动
- 便于独立部署、独立扩展
- 利于面试展示（API 可直接用 Postman 演示）

## ADR-002: 用户体系延后实现

**背景**：原平台用户体系开发中，规范未定。

**决策**：MVP 阶段使用匿名 session（cookie 中存 UUID），所有 API 接口
预留 `user_id` 字段。未来对接时实现 `AuthAdapter` 即可接入平台鉴权。

## ADR-003: LLM 选择 DeepSeek

**背景**：需在效果、成本、API 稳定性间权衡。

**决策**：MVP 阶段使用 DeepSeek V3 API。

**理由**：成本远低于 GPT-4 / Claude（中文场景效果接近），月预算可控在百元级。

**替代方案**：未来可在 `llm/` 模块替换为 OpenAI / Claude / 本地模型，
所有调用集中在 `LLMClient` 抽象。

## ADR-004: 向量库选择 Chroma

**背景**：需要本地可运行、易部署的向量库。

**决策**：使用 Chroma，文件持久化模式（不跑独立服务）。

**理由**：零运维、Python 原生、对 90 篇论文规模完全够用。

**替代方案**：用户量增长后可迁移到 Qdrant / Milvus。

## ADR-005: Agent 框架自研而非 LangChain

**背景**：LangChain 抽象重、调试难，对学术研究项目不友好。

**决策**：基于原生 OpenAI 兼容协议自研轻量 Agent 框架，
ReAct 循环 < 200 行代码，所有 prompt 可见、可控、可改。

**理由**：
- 论文写作需要清晰描述算法细节，黑盒框架不利于阐述
- 调试 Agent 决策路径时，自研代码透明度远高于 LangChain
- 90 篇论文 + 单 Agent 场景不需要复杂编排

## ADR-006: WHCNS 接入策略

**背景**：WHCNS 是黑盒可执行模型，输入输出已确认。

**决策**：在 `simulator/` 模块用 mock 实现先占位，定义清晰的接口契约，
真实接入推迟到 M3 阶段。

**接口契约**：
- 输入：标准化的农田参数 dict（土壤、气象、施肥方案）
- 输出：氮淋失量时间序列 + 关键指标
- 调用：异步任务（仿真耗时不可忽略）

## 里程碑

- **M1（5 月）**：基础对话闭环 — 用户可在 Web 上问答，回答带引用
- **M2（6 月）**：完整 Agent 能力 — 多轮、工具调用、流式输出、对话历史
- **M3（7 月）**：WHCNS 集成 + 工作流模式
- **M4（8 月）**：评测完善 + 上线 + 演示打磨
